▶ How do I get my API token?
  1. Log in to manage.mist.com
  2. Click your name in the top-right corner → My Profile
  3. Scroll down to API Token and click Create Token
  4. Give it a name (e.g. "Deny Log Report") and click Generate

  6. Paste it in the field below
Required access: The token needs org-level read access to NAC client events and sites. An org admin token works.

What is this tool?

The Deny Log Intelligence dashboard transforms raw RADIUS deny events from Mist Access Assurance into a client-centric view. Instead of seeing hundreds of identical rows for a single broken device, you get one record per client — showing exactly why it's failing, how long it's been failing, and who else is hitting the same issue.

Metric Cards (top row)

CardWhat it means
Total Deny EventsThe raw number of individual RADIUS deny events from the Mist API in the selected time window. One broken client retrying 50 times = 50 events.
Unique ClientsThe number of distinct devices (by MAC address) that have at least one deny event. This is the number that actually matters for triage.
Sites AffectedHow many distinct Mist sites have at least one failing client.
Cert FailuresClients whose primary failure reason is a certificate or TLS trust issue — typically the most common and most fixable category.
Silent FailuresClients that failed but have not retried in 8+ business hours (Mon–Fri, 7am–7pm). These are invisible in real-time dashboards. See Status below for full definition.

Failure Categories

CategoryMeaningCommon fix
🔐 Cert / TLS IssueThe device doesn't trust the RADIUS server certificate, or its own client certificate is missing, expired, or from an untrusted CA.Deploy the Mist org certificate to the device via MDM. Go to Organization → Access → Certificates to download it.
🔑 Wrong CredentialsThe username or password was rejected by the identity provider (IdP). The device itself may be fine.Verify credentials, check account is not locked, confirm the user's IdP group grants network access.
📱 MAC Auth / Policy FailureThe device hit the implicit deny — no NAC policy rule matched it. Often a managed device not yet enrolled, or a policy misconfiguration.Confirm the MAC is in the correct enrollment list. Check that a NAC rule exists for this device type or SSID.

Client Status

StatusDefinitionAction
failingThe client has a deny event within the last 8 business hours. It is actively trying and being rejected right now.Immediate — the user is likely on-site and unable to connect.
silentThe client's last deny event was more than 8 business hours ago (Mon–Fri, 7am–7pm) and no successful auth has been seen. It stopped retrying — the user may have given up or is working from home. Weekend hours are excluded, so a Friday failure won't flip to silent until Monday.Proactive — reach out before the user files a ticket on Monday morning.
resolvedA successful RADIUS permit event was seen for this client within the lookback window, after the deny events.No action needed. Monitor to confirm it doesn't reappear.

Blast Radius

Blast radius measures how many clients share the exact same deny reason text. A high blast radius means a systemic issue — for example, a certificate that wasn't deployed to all managed Windows devices, causing 20 machines to fail with the same error message.

ScoreThresholdWhat it suggests
HIGH5 or more clients with the same deny reasonSystemic — likely a policy change, failed MDM deployment, or cert rollout that missed a fleet of devices.
MED3–4 clients with the same deny reasonSmall group pattern — possibly a shared policy, same device model, or same SSID configuration issue.
LOW1–2 clients with this deny reasonIsolated — specific to that device. Likely a misconfigured endpoint or user account issue.

Deny Reasons Tab

This tab lists every unique error message returned by the RADIUS server, sorted by how many clients are hitting it. Click any row to jump to the Dashboard tab filtered to only those clients. This is the fastest path to triage: see the most impactful error first, then act on it.

7-Day Timeline

Shows a row per client with colored blocks for each day they had deny events. This makes it easy to spot:

  • Persistent failures — colored blocks spanning the full 7-day window
  • Sudden onset — all clients start failing on the same day (likely a policy or cert change)
  • Silent failures — colored blocks only on early days, nothing recent

Colors match the category: ■ orange = cert · ■ purple = credentials · ■ teal = MAC/policy

Notification Center

Pre-written emails ready to send. Select a site to see an email addressed to that site's building tech or IT contact, pre-populated with the affected MACs, deny reasons, and step-by-step remediation instructions. Edit as needed before sending. The NOC Digest covers all sites in a single summary email.

Frequently Asked Questions

Why does my client show as "silent" even though it failed today?Silent requires 8+ business hours without a retry. If the failure happened today but earlier in the day, it may not have crossed the threshold yet.
A client shows "resolved" but the user says they still can't connect.Resolved means a permit was seen in the lookback window — but it may have failed again after. Try a shorter lookback window (3 days) to see only recent events.
I see the same MAC with different usernames.That's normal for shared devices or devices that re-authenticate under different accounts. The MAC is the primary key; usernames are shown for reference.
The report shows 0 clients but I know there are failures.Check that your token has org-level (not site-level) access. Site-level tokens can only see one site and may miss cross-site events.
Can I extend the lookback beyond 14 days?Event retention in Mist varies by org tier. If you request 14 days but your org only retains 7, you'll see 7 days of data with no error.
